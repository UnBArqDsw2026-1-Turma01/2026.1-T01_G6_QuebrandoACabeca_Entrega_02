## 2. Metodologia
Realizamos uma reunião onde consolidamos as funcionalidades e o escopo do projeto no diagrama de classes, utilizando o Miro como ferramenta colaborativa para mapear as conexões do sistema. O modelo reflete a visão do grupo ao integrar o gerenciamento de usuários e autenticação à lógica central do jogo.
Utilizamos os slides e aulas da professora Milene Serrano como referência, além das informações no website [uml-diagrams](https://www.uml-diagrams.org/class-diagrams-overview.html). A ata pode ser encontrada em [Ata Reunião](../AtasReunioes/Ata1.md). 

Depois tivemos que fazer uma revisão assíncrona, acompanhada pelo Whatsapp mesmo, devido a dificuldade de horários e porque no diagrama tinham atributos redundantes que já eram representados pelas relações. 

Além disso, basicamente para a identificação dos elementos do diagrama, adotamos uma abordagem empírica de análise de domínio baseada em brainstorming, nos comunicando. A equipe mapeou as classes, atributos e relacionamentos debatendo iterativamente as entidades fundamentais do escopo (como partidas, galerias, configurações e usuários) diretamente na ferramenta Miro. Em vez de uma metodologia formal rígida, optamos pela extração colaborativa a partir dos requisitos e funcionalidades desejadas, refinando o modelo de forma até alcançar coesão necessária para representar as estruturas do sistema.

## 3. Justificativa
Escolhemos elaborar o diagrama de classes por ser a ferramenta ideal para descrever a estrutura de um sistema, permitindo fazer em um modelo visual que define como os componentes serão construídos. Optamos por esse artefato para que a equipe tivesse clareza sobre as responsabilidades de cada parte do software, separando corretamente os dados estáticos da lógica de execução das partidas.

Conforme destacado pelo Lucidchart (2026), o diagrama de classes UML é um dos tipos mais úteis de diagramas de estrutura, pois mapeia de forma clara a composição de um sistema ao detalhar suas classes, atributos, operações e as relações entre os elementos. Ao documentar atributos e métodos de forma técnica, validamos a viabilidade do que foi discutido anteriormente e estabelecemos um roteiro objetivo para o desenvolvimento.

Além disso, o diagrama de classes foi escolhido em vez dos diagramas de componentes ou de implantação porque a complexidade central do nosso software reside nas relações entre as entidades de domínio e lógicas de estado. O sistema depende de estruturas de dados interligadas: Usuários, Galerias (para salvar imagens e progressos), Configurações de Dificuldade (rotação, tempo, tipo de encaixe), Partidas (singleplayer e 1x1 multijogador), Publicações sociais e Conquistas.

Modelar estaticamente essas classes, seus métodos e multiplicidades (ex: uma partida 1x1 atrelada a dois jogadores e afetando um ranking global; um usuário possuindo múltiplos quebra-cabeças pausados na galeria) é essencial para garantir a integridade estrutural do banco de dados e da lógica de negócios. A prioridade técnica para este escopo é consolidar as regras do jogo e do ecossistema social na estrutura do código, tornando a modelagem de classes mais relevante do que mapear a arquitetura física de servidores ou o empacotamento de módulos, então concluímos que esses outros seriam diagramas simples que não trariam valor tão relevante para nosso projeto.

## 4. Senso Crítico
O diagrama de classes mostra ser um artefato muito bom por converter a visão geral do projeto em uma estrutura técnica funcional, atendendo plenamente à expectativa de organizar o escopo em componentes lógicos e integrados. Sua utilidade é bem clara ao definir rigorosamente as relações entre usuários, mecânicas de jogo e o processamento de imagens, eliminando ambiguidades e servindo como um guia seguro para a implementação do banco de dados e do código. O modelo atual  estabelece uma base sólida, coesa e capaz de suportar o desenvolvimento das funcionalidades discutidas no brainstorming e sugestões de funcionalidades dadas pela professora posteriormente.

Contudo, o modelo apresenta limitações. A lógica de interação do usuário (arrastar, encaixar e rotacionar peças) é voltada para a interface e eventos físicos, tornando-se bem abstrata em um diagrama de classes. Além disso, o diagrama não capta a complexidade da sincronização em tempo real e concorrência exigida no modo de partidas 1x1, apesar que nesse caso pretendemos apenas sincronizar os tempos e o progresso dos usuários. Mas, o modelo é útil para estruturar o banco de dados das galerias e histórico, com a junção do diagrama dinâmico e o de casos de uso fecha um trio que representa bem o projeto.

## 5. Rastreabilidade & Elos com Outros Artefatos

O Diagrama de Classes desempenha o papel de espinha dorsal estrutural da modelagem do projeto, fornecendo o vocabulário de entidades, atributos e operações que sustentam tanto o comportamento descrito na [Modelagem Dinâmica](2.2.ModelagemDinamica.md) quanto as funcionalidades elicitadas na [Modelagem Organizacional / Casos de Uso](2.3.ModelagemOrganizacionalCasosDeUso.md). Em outras palavras, cada caso de uso encontra aqui suas classes responsáveis, e cada interação dos diagramas de sequência manipula instâncias destas classes.

A tabela abaixo consolida os principais elos de rastreabilidade entre os três artefatos:

| Classe (2.1) | Caso de Uso correspondente em [2.3](2.3.ModelagemOrganizacionalCasosDeUso.md) | Diagrama de Sequência correspondente em [2.2](2.2.ModelagemDinamica.md) |
| -- | -- | -- |
| `Autenticador`, `Usuario` (operações `criarUsuario`, `loginUsuario`, `recuperarSenha`) | *Gerenciar Conta* (módulo Jogo, ator externo *Authentication*) | DS2 — Editar dados da conta (`ContaController` → `ContaService` → `ContaRepository`) |
| `Imagem` (operações `uploadImagem`, `excluirImagem`) e a associação `ImagensEnviadas`/`ImagensFavoritas` com `Usuario` | *Obter Imagem* e *Gerenciar Favoritos* (módulo Galeria) | DS6 — Envio de imagem (`ImagemController` → `ImagemService` → `ImagemRepository`) |
| `QuebraCabeca` e `Peca` (operações `iniciar`, `processarImagem`, `gerarPecas`, `randomizarLocalizacaoPecas`, `mover`) | *Criar Quebra-Cabeças* (módulo Galeria) e *Selecionar Quebra-Cabeças para Jogar* (módulo Jogo) | DS1 — Criar quebra-cabeça (`PuzzleController` → `PuzzleService` → `PuzzleRepository`) |
| `Partida`, `QuebraCabeca`, `Peca` (operações `criarQuebraCabeca`, `contagemTempo`, `salvarProgresso`) | *Jogar Quebra-Cabeça Individualmente* + extensão *Salvar Progresso* | DS3 — Iniciar jogo / mover peça / salvar progresso (`JogoController` → `JogoService` → `JogoRepository`) |
| `Partida` com multiplicidade `participantes (1..2)` ligando dois `Usuario` | *Jogar Partida 1x1* (atores Usuário e Oponente) | DS4 — Entrar em partida online (`OnlineController` → `OnlineService` → `SalaRepository`) |
| `Publicação` (operações `criarPublicacao`, `editarPublicacao`, `postarPublicacao`, `excluirPublicacao`) e agregação com `Imagem` | *Gerenciar Publicações* + extensão *Anexar Conteúdo* (módulo Social) | DS5 — Nova publicação (`PublicacaoController` → `PublicacaoService` → `PublicacaoRepository`) |
