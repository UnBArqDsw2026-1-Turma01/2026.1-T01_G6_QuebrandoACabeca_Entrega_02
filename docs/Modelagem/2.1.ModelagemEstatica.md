## 2. Metodologia
Realizamos uma reunião onde consolidamos as funcionalidades e o escopo do projeto no diagrama de classes, utilizando o Miro como ferramenta colaborativa para mapear as conexões do sistema. O modelo reflete a visão do grupo ao integrar o gerenciamento de usuários e autenticação à lógica central do jogo.
Utilizamos os slides e aulas da professora Milene Serrano como referência. a ata pode ser encontrada em [Ata Reunião](../AtasReunioes/Ata1.md)


## 3. Justificativa
Escolhemos elaborar o diagrama de classes por ser a ferramenta ideal para descrever a estrutura de um sistema, permitindo fazer em um modelo visual que define como os componentes serão construídos. Optamos por esse artefato para que a equipe tivesse clareza sobre as responsabilidades de cada parte do software, separando corretamente os dados estáticos da lógica de execução das partidas.

Conforme destacado pelo Lucidchart (2026), o diagrama de classe UML é um dos tipos mais úteis de diagramas de estrutura, pois mapeia de forma clara a composição de um sistema ao detalhar suas classes, atributos, operações e as relações entre os elementos. Ao documentar atributos e métodos de forma técnica, validamos a viabilidade do que foi discutido anteriormente e estabelecemos um roteiro objetivo para o desenvolvimento.

## 4. Senso Crítico
O diagrama de classes mostra ser um artefato muito bom por converter a visão geral do projeto em uma estrutura técnica funcional, atendendo plenamente à expectativa de organizar o escopo em componentes lógicos e integrados. Sua utilidade é bem clara ao definir rigorosamente as relações entre usuários, mecânicas de jogo e o processamento de imagens, eliminando ambiguidades e servindo como um guia seguro para a implementação do banco de dados e do código. O modelo atual  estabelece uma base sólida, coesa e capaz de suportar o desenvolvimento de todas as funcionalidades discutidas no brainstorming.
