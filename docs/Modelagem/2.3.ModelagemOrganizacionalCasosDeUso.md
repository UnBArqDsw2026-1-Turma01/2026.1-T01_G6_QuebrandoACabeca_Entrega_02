## 2. Metodologia

O diagrama foi feito pelo Grupo 6 em uma sessão no Miro, onde conseguimos trabalhar juntos no mesmo quadro ao mesmo tempo, o que ajudou bastante na construção colaborativa, nesse caso fizemos também de forma assíncrona mas se comunicando pelo grupo do Whatsapp.
 
Seguimos a notação da UML 2.5 como referência, slides da professora e o site [uml-diagrams](https://www.uml-diagrams.org/class-diagrams-overview.html). De forma geral, passamos pelas seguintes etapas:
 
1. Levantamento de requisitos: revisamos o escopo do projeto para identificar o que o sistema precisa fazer e quem vai usar.
2. Identificação de atores chegamos à conclusão que o ator principal é claramente o Usuário Final já que as funcionalidades não dependem de atores externos, exceto a autenticação que representamos como um ator serviço e o oponente que é outro usuário.
3. Elicitação dos casos de uso: mapeamos as funcionalidades dos dois escopos do projeto: a geração de imagens em formato quebra-cabeça a partir de fotos, e a montagem virtual interativa, além das funcionalidades de suporte: autenticação, parte da galeria e a parte social do software.
4. Revisão coletiva: ao final, o grupo revisou o diagrama junto para garantir que nada ficou de fora.

## 3. Justificativa

Escolhemos o diagrama de casos de uso por ser um dos artefatos mais acessíveis para representar o que o sistema faz sem precisar entrar em detalhes de como ele faz, o que faz sentido nessa fase do projeto, onde ainda estamos definindo comportamentos antes de tomar decisões de implementação. Ele pode delimitar as fronteiras do sistema e mapear claramente as interações do usuário final com as múltiplas frentes do software.

E ele serve como base contratual para a elicitação dos requisitos funcionais, orientando o desenvolvimento das próximas etapas sem a necessidade de detalhar a implementação técnica neste momento.

Além disso descartamos o Diagrama de Pacotes porque a arquitetura do software ainda não está madura o suficiente para ser subdividida em módulos de implementação. A prioridade técnica atual é mapear e validar todas as ações do jogador, desde o fluxo social até o multijogador, o que o Diagrama de Casos de Uso atende diretamente. Estruturar pacotes agora seria um agrupamento prematuro de código que agregaria baixo valor técnico nesta etapa de concepção.

## 4. Senso Crítico

No geral, achamos que o artefato cumpriu bem o que a gente esperava dele. O processo de montar o diagrama em si foi útil porque nos obrigou a sentar e definir de forma mais clara o que o sistema precisa fazer, e algumas coisas que pareciam óbvias no início geraram bastante discussão na prática.
 
O artefato é eficiente para organizar as funcionalidades de CRUD e interações sociais (galeria, publicações, favoritos e histórico de rankings), basicamente as funcionalidades principais do usuário que virarão as páginas no website, mas é falho ao tentar modelar a complexidade da jogabilidade.
 
A principal limitação está na abstração excessiva das mecânicas de jogo: interações granulares de UI (rotacionar, mover e encaixar peças) não podem ser mapeadas aqui sem poluir o modelo. Além disso, o diagrama de casos de uso não captura a dinâmica de concorrência e a sincronização de rede exigidas pelas partidas multijogador 1x1 em tempo real.
 
O diagrama de casos de uso mostra o quê, mas não o como. Para um sistema que envolve processamento de imagem, é necessário o complemento com diagramas de sequência para detalhar os fluxos mais complexos.

Não modelamos fluxos alternativos nem de exceção. O que o sistema faz se a foto enviada for inválida, por exemplo, ainda não está representado. O modelo falha em demonstrar como o sistema reage a desconexões de um dos jogadores ou como os tempos são validados para o ranking.

De forma geral, o artefato atendeu as expectativas para essa etapa, mas ficou claro que ele sozinho não é suficiente para documentar tudo que o sistema precisa fazer.

## 5. Rastreabilidade & Elos com Outros Artefatos

O Diagrama de Casos de Uso atua como ponto de entrada da modelagem do projeto: ele descreve o quê o sistema faz e para quem, servindo de referência para a [Modelagem Estática](2.1.ModelagemEstatica.md), que apresenta as classes responsáveis pela realização desses casos de uso, e para a [Modelagem Dinâmica](2.2.ModelagemDinamica.md), que detalha as interações entre os objetos durante a execução de cada cenário.

A tabela a seguir mostra como cada caso de uso identificado neste diagrama é relacionado nos demais artefatos:

| Módulo | Caso de Uso (2.3) | Realização estrutural em [2.1](2.1.ModelagemEstatica.md) | Realização comportamental em [2.2](2.2.ModelagemDinamica.md) |
| -- | -- | -- | -- |
| **Jogo** | *Gerenciar Conta* (com *Authentication*) | Classes `Autenticador` e `Usuario` (operações `criarUsuario`, `loginUsuario`, `recuperarSenha`) | DS2 — Editar dados da conta (`ContaController` → `ContaService` → `ContaRepository`) |
| **Jogo** | *Jogar Quebra-Cabeça Individualmente* + *Selecionar Quebra-Cabeças para Jogar* + extensão *Salvar Progresso* | `Partida`, `QuebraCabeca` (`iniciar`, `finalizar`), `Peca` (`mover`, `estaMontado`), `Usuario` (`salvarProgresso`) | DS3 — Iniciar jogo / mover peça / salvar progresso (`JogoController` → `JogoService` → `JogoRepository`) |
| **Jogo** | *Jogar Partida 1x1* (atores Usuário e Oponente) | Classe `Partida` com associação `participantes (1..2)` ligando dois `Usuario` | DS4 — Entrar em partida online (`OnlineController` → `OnlineService` → `SalaRepository`) |
| **Jogo** | *Visualizar Ranking* / *Visualizar Conquistas* | Atributo `pontuacao` em `Partida` e operação `resultado()` que retorna `Usuário` | *(comportamento ainda não detalhado nesta entrega — candidato a novo diagrama de sequência)* |
| **Galeria** | *Criar Quebra-Cabeças* (include *Obter Imagem*) | Associação `quebraCabecasCriados (0..*)` entre `Usuario` e `QuebraCabeca`; operações `processarImagem`, `gerarPecas` | DS1 — Criar quebra-cabeça (`PuzzleController` → `PuzzleService` → `PuzzleRepository`) |
| **Galeria** | *Obter Imagem* — especialização *Fazer Upload de Imagem* / *Selecionar Imagem Salva ou Favoritada* | Classe `Imagem` (`uploadImagem`, `excluirImagem`) e associações `ImagensEnviadas` / `ImagensFavoritas` com `Usuario` | DS6 — Envio de imagem (`ImagemController` → `ImagemService` → `ImagemRepository`) |
| **Galeria** | *Gerenciar Favoritos (Imagens / Quebra-Cabeças)* | Operação `favoritarImagem` em `Usuario` e associação `ImagensFavoritas (0..*)` | *(reaproveita DS6 e fluxos da Galeria)* |
| **Social** | *Gerenciar Publicações* + extensão *Anexar Conteúdo* | Classe `Publicação` (`criarPublicacao`, `editarPublicacao`, `postarPublicacao`, `excluirPublicacao`) com agregação a `Imagem` e associação a `Usuario` | DS5 — Nova publicação (`PublicacaoController` → `PublicacaoService` → `PublicacaoRepository`) |
 
## 6. Comentários sobre o Trabalho

O Miro já é uma ferramenta familiar para o grupo, então não tivemos dificuldade de adaptação. A escolha se mostrou adequada para a sessão de modelagem colaborativa, já que conseguimos construir e ajustar o diagrama juntos em tempo real.

A maior dificuldade foi decidir a granularidade dos casos de uso. Ficamos um tempo discutindo até onde detalhar cada funcionalidade: casos muito amplos ficam vagos e pouco úteis, mas casos muito detalhados acabam poluindo o diagrama. No fim, optamos por um nível intermediário, priorizando a legibilidade.

Para as próximas entregas, seria interessante registrar minimamente as decisões tomadas durante as sessões, mesmo que de forma informal, para facilitar a rastreabilidade do que foi discutido e do porquê de certas escolhas.
